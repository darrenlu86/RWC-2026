## Abstract

本研究探討如何推動隱私聚合（private aggregation）技術的大規模採用，使其像 TLS 一樣普及。論文即將發表於 USENIX Security 2026。核心觀察是：現行雙伺服器隱私聚合方案（如 Prio、Whisper）要求第二伺服器承擔與應用伺服器相當的工作量，導致需付費外包運營，形成採用障礙——正如 Let's Encrypt 出現前付費憑證阻礙了 TLS 普及。研究者提出「Heli」（Heavy-Light）聚合方案，將第二伺服器（light server）的工作量與通訊量降至與客戶端數量無關的常數級別，使非營利組織免費運行 light server 成為可行方案。在一千萬模擬使用者的端到端實驗中，Heli 的 light server 成本比現有方案低最高 2,300 萬倍。

## Summary

### 隱私聚合的動機與問題定義

開發者為了改善產品，需要了解應用程式的使用狀況。以 WhatsApp 為例，它可能想知道「今天有多少比例的使用者封鎖了聯絡人」。目前的做法是直接從每個使用者收集一個 bit，指示是否封鎖了某人。問題在於 WhatsApp 的出發點是群體級別的統計問題，但實際上卻學到了每個個別使用者的具體行為。

更進一步，WhatsApp 可能會問更具體的問題，例如「今天有多少比例的青少年使用者封鎖了聯絡人」，這就同時收集了行為資料和人口特徵。實務上，服務商會問許多不同的問題（是否啟用限時訊息？旅行時使用量是否增加？），這些資料組合起來會構建出非常詳細的使用者側寫，包含敏感資訊。對於 WhatsApp 這類通訊應用，服務商還知道使用者的社交關係，因此可以進行跨使用者的關聯分析。

**隱私聚合（Private Aggregation）** 以原始使用者資料為輸入，僅輸出聚合結果（加總）。雖然這是一類簡單的量測方式，但可以用來計算直方圖、模型更新等。它具備兩個核心屬性：
1. **隱私性（Privacy）**：即使伺服器是惡意的，也只能學到聚合結果，無法得知個別客戶端資料
2. **正確性（Robustness）**：能偵測並排除提交無效報告的惡意客戶端——例如某客戶端本應提交一個 bit，卻嘗試提交一個極大數值來偏斜結果，系統可以偵測並移除該使用者

### 從 TLS 採用歷程汲取教訓

講者以 2014-2019 年 TLS 採用率從約 30% 頁面載入量躍升至 80% 的成功故事為類比，歸納出兩大驅動力：

1. **免費憑證**：2014 年以前，憑證需要付費購買。付費不僅是金額問題，更重要的是流程摩擦——開發者需要向外部企業付款，可能要找主管簽核、涉及商業合約與律師，整個過程變得複雜。Let's Encrypt 於 2015 年成立並提供免費憑證，消除了這個障礙。
2. **平台壓力**：瀏覽器開始對未使用 TLS 的網站顯示 UI 警告，Google 等搜尋引擎降低未加密網站的搜尋排名。

核心啟示：**讓技術免費取得，並透過外部激勵推動採用。**

### 現行雙伺服器模型的瓶頸

目前經過數年標準化（超過五年）的隱私聚合方案（Prio、Whisper）需要兩台伺服器，所有客戶端同時與兩台伺服器通訊。為了保障隱私，兩台伺服器必須由不同實體運營——WhatsApp 不能自己運行第二台伺服器。目前的解決方案是付費外包第二台伺服器，但付費服務正是阻礙 TLS 普及的相同因素。

已有豐富的研究嘗試消除第二伺服器，改由客戶端子集執行多方計算（MPC）來模擬第二伺服器（如 Lerna、Opa 等）。但講者指出，這類方案存在一個關鍵取捨：**必須在「對抗惡意伺服器的隱私性」和「對抗惡意客戶端的正確性」之間二擇一**，目前沒有同時達成兩者的方案。許多應用需要同時具備兩個屬性，因此講者聚焦於雙伺服器模型。

### Heavy-Light 部署模型

仿效 Let's Encrypt，理想方案是由非營利組織免費運行第二台伺服器。但在聚合運算中，第二伺服器的工作量與應用伺服器相當——即使只有數千個應用各有數百萬客戶端，這個工作量對非營利組織而言也無法承受。

因此提出 **Heavy-Light Aggregation** 模型：
- **Heavy server**（由應用提供商運營）：與所有客戶端通訊，承擔主要計算負載
- **Light server**（由非營利組織運營）：**完全不與客戶端通訊**，工作量與通訊量為**常數**，與客戶端數量無關
- **一次性設定（One-time setup）**：允許 light server 在協議開始時與所有客戶端通訊一次。假設每個客戶端會隨時間參與多次量測，因此可攤提這次設定的成本
- 此模型的一個放寬是：對抗惡意 heavy server 的保護比傳統雙伺服器模型更困難，因此需要一次性設定來建立客戶端集合的承諾

### 技術核心：Heli 協議的構建

Heli 協議由三個組件組成：(1) 基礎協議、(2) 標準零知識證明（讓客戶端證明行為正確）、(3) 客戶端在設定階段驗證身分的機制。講者強調這是對現有方案的一個「超級簡單的變形（super simple twist）」，並使用橢圓曲線密碼學實例化。講者認為有可能將底層原語替換為後量子安全的版本，但尚未正式證明。

構建過程分三步：

**第一步：基礎 secret sharing 方案。** 客戶端將輸入編碼到某個群中，並將其 secret share 給應用伺服器和第二伺服器。兩台伺服器各自對收到的 share 求和，輸出聚合結果。在多次統計量測中，第二伺服器的工作量隨統計次數和客戶端數量線性增長。

**第二步：以 PRF 降低通訊量。** 客戶端在進行 secret sharing 時，使用偽隨機函數（PRF）產生的偽隨機值替代真隨機值。如此一來，客戶端不再需要將所有隨機值傳送給第二伺服器，只需傳送一個 PRF key。第二伺服器自行評估 PRF 即可。這降低了通訊量，但計算量不變。

**第三步：以 Key Homomorphic PRF 降低計算量。** Key Homomorphic PRF 具有標準 PRF 的安全性，但額外擁有 key homomorphism 特性：

> F(k1, x) + F(k2, x) = F(k1 + k2, x)

即同一輸入上對不同 key 的 PRF 輸出之和，等於在 key 之和上的單次 PRF 輸出。具體構造非常簡潔：取一個到 DDH 困難群的雜湊函數 H，計算 H(x)^k 即為一個 key homomorphic PRF。

利用此特性，第二伺服器原本需要計算所有客戶端 PRF 輸出之和，現在等價於對 key 之和的單次 PRF 評估。由於 key 之和在多次統計量測間共享，只需在設定階段計算一次。此後每輪聚合，light server 只需進行**一次 PRF 評估**。

### 實驗評估

使用 Rust 實作，基於 **Ristretto 255 群**，與 Prio 和 Whisper（兩個經歷標準化的方案）進行比較。在 AWS 上以一千萬模擬客戶端進行端到端評估，每個客戶端提交 32 個 1-bit 輸入：

| 指標 | Prio/Whisper | Heli |
|------|-------------|------|
| Heavy server 與 Light server 成本 | 兩者相當 | 嚴重不對稱 |
| Light server 相對成本 | 基準 | 低最高 **2,300 萬倍** |
| Heavy server 相對成本 | 基準 | 高最多 **60 倍** |
| 整體端到端（32-bit, 10M 客戶端） | — | Heavy server 貴 38 倍，Light server 便宜 12 萬倍 |

- Heavy server 的成本增加瓶頸完全來自**驗證客戶端零知識證明**，底層證明系統的任何改進都會直接降低此成本
- 在 happy path（無客戶端退出）下，light server 的具體成本為**一次指數運算**，通訊**一個群元素**

### 目標達成與限制條件

講者認為目標大致達成，但有以下附帶條件：
1. **需要一次性設定**：light server 必須在開始時與所有客戶端通訊一次
2. **需要穩定的客戶端群體**：若有客戶端在設定後退出，heavy server 必須通知 light server 哪些客戶端未參與本輪聚合，這會產生額外成本
3. **應用提供商承擔更高內部成本**：heavy server 成本顯著增加

核心結論：**Heli 讓免費的第二伺服器在實務上變得可行。**

### 採用的非技術障礙與激勵機制

即使技術問題完美解決，隱私聚合仍缺乏經濟誘因：相較於標準遙測，隱私聚合成本更高、分析彈性更低、且無法利用資料進行廣告投放或模型訓練。從純經濟角度，幾乎沒有理由主動採用。

TLS 面臨過相同困境——運行 TLS 的成本更高，企業缺乏動力。推動採用的是**平台壓力**：瀏覽器對未加密網站顯示警告、搜尋引擎降低排名。對隱私聚合而言，最可能的推動者是 **iOS 和 Android 平台**，在應用商店中對使用隱私聚合的應用給予激勵或要求。講者設想的理想狀態是：平台對離開裝置的資料提供更多透明度，讓使用者了解應用程式正在匯出哪些類型的資訊，若應用使用隱私聚合，則在相關項目旁顯示一個鎖頭圖示（類似 TLS 的視覺指標）。

### 開放問題

1. **客戶端退出的常數開銷**：能否在客戶端退出時維持 light server 常數級工作量，同時不大幅增加 heavy server 的負擔？
2. **加速證明驗證**：能否在 heavy-light 設定下加速客戶端零知識證明的驗證？Heavy server 的所有開銷都來自證明驗證，而 light server 完全不參與該過程——是否可以設計新的證明系統讓 light server 協助 heavy server？
3. **資料匯出的 UI 等價物**：TLS 的成功部分歸功於瀏覽器的 UI 警告，對於資料匯出場景，什麼樣的 UI 設計能達到同等效果？講者認為這是一個更複雜的問題。
4. **後量子安全性**：理論上可以用格基（lattice-based）key homomorphic PRF 替換當前的橢圓曲線構造，但尚未正式證明安全性。

### Q&A 重點摘錄

**關於 ElGamal 加密替代方案（電子投票技術）**：有人提問為何不讓每個客戶端直接用 ElGamal 加密並附上零知識證明，heavy server 透過乘法聚合密文，light server 只做一次解密。講者回應指出，此方案的問題在於 heavy server 對哪些客戶端被納入聚合擁有完全控制權——它可以丟棄所有其他提交，只保留單一客戶端的密文交給 light server 解密，從而破壞隱私。Heli 的一次性設定正是為了解決此問題：它建立了「我要聚合的客戶端集合」的承諾，確保聚合確實在約定的客戶端集合上進行。

**關於差分攻擊（delta attack）**：有人擔心惡意運營者可以先請求包含所有使用者的聚合結果，再請求排除目標使用者的聚合結果，透過差值推斷個人資料。講者解釋：每個客戶端的提交都綁定到特定的一輪（round），只要兩台伺服器中有一台是誠實的，就無法將同一客戶端的提交用於多輪聚合。客戶端的 secret sharing 綁定特定上下文（context），light server 只會對該特定上下文進行解密。

## 與 TWCA 臺灣網路認證的關聯

本場演講與 TWCA 的業務沒有直接關聯，但有多個值得關注的間接啟示：

1. **Let's Encrypt 模式的深層借鑑**：本研究直接以 Let's Encrypt 的成功作為隱私聚合推廣的藍本，核心理念是「免費公共基礎設施推動技術普及」。TWCA 作為臺灣的憑證機構，親身經歷了 Let's Encrypt 對傳統 CA 商業模式的衝擊，應深入思考這種模式是否會延伸到隱私保護領域。若未來出現「隱私聚合版的 Let's Encrypt」非營利組織，TWCA 可評估是否主動參與或轉型。

2. **Light Server 角色的信任服務機會**：Heavy-Light 模型要求 light server 由獨立於應用提供商的可信第三方運營。TWCA 作為具備公信力的第三方信任機構，在技術成熟後有天然優勢扮演 light server 角色，為臺灣企業提供隱私聚合的信任基礎設施服務。

3. **隱私法規趨勢下的新服務方向**：隨著全球隱私法規趨嚴（GDPR、臺灣個資法修訂），企業對「既能收集有用遙測數據，又能證明保護使用者隱私」的技術需求將持續增長。隱私聚合正是這類需求的技術解方，TWCA 可關注此領域的標準化進展與商業化時機。

4. **雙伺服器信任模型與 CA 角色的延伸**：隱私聚合系統要求兩台伺服器由不同實體運營以確保隱私，這種「獨立第三方信任保證」與 TWCA 現有的 CA 角色高度一致。未來若平台（如 iOS/Android）要求應用使用隱私聚合並需要認證合格的 light server 運營商，CA 機構將是天然候選者。
