## Abstract

SecureDrop 是開源匿名吹哨系統，歷經十年部署後面臨架構老化問題。本演講介紹下一代 SecureDrop 協議設計，針對雲端部署場景重新設計加密協議，採用 HPKE 實現混合（後量子+經典）機密性保障，同時維持來源端無狀態、無用戶表、協議層可否認性等嚴格隱私需求，並透過計算證明與 Tamarin 符號模型進行雙重安全分析。

## Summary

### SecureDrop 現有架構與限制

SecureDrop 允許吹哨者透過 Tor 匿名聯繫新聞編輯室。現有架構已運行十年，核心設計未變：
- 新聞組織在內部部署實體伺服器
- 來源透過 Tor Browser 存取 onion service 上的 Web 應用
- 檔案和訊息在伺服器端使用 PGP 加密
- 記者在離線環境中下載並解密

新聞組織保留基礎設施控制權，可直接對抗執法機關的資料索取（如傳票）。來源不需執行 JavaScript，也不需下載專用應用程式。

然而，現有架構有明顯缺點：
- 系統管理員責任繁重（部署、維護、金鑰管理等）
- 必須確保伺服器所在位置的實體安全
- 對於可能遭遇突擊搜索的地區，不建議部署
- 對小型調查團隊和獨立記者來說，自行託管實體基礎設施門檻過高

### 下一代設計需求

下一代 SecureDrop 的需求極為嚴格：

1. **首次接觸（First Contact）**：非同步、無需額外通道即可引導金鑰材料，且來源不應公開發布自己的公鑰

2. **端對端加密**：為基本要求，同時追求不可觀察性（unobservability）、不可連結性（unlinkability）和匿名性（anonymity）

3. **易於部署**：支援雲端部署以降低成本和門檻，但雲端部署大幅擴大攻擊面，需要嚴格的威脅建模

4. **極高的元資料標準**：沒有用戶表、沒有帳號概念、沒有時間戳記——甚至顯示當前活躍來源數量都被視為不可接受的元資料洩露

5. **不惡化來源處境**：不在來源裝置上寫入檔案；考慮協議層可否認性，避免產生不當的歸罪證據

### 為何不採用現有方案

- **Signal 協議**：需要雙方保持客戶端狀態，不符合來源無狀態需求
- **MLS**：同樣需要客戶端狀態，且為不可否認協議
- **PIR（私有資訊檢索）**：限制條件的組合使其暫時不可行

團隊引用的核心觀點：可否認性不是一張「免罪金牌」，而是避免產生額外歸罪證據——這在當前美國等國家的執法環境下，已非學術假設。

### 新協議架構

**參與者**：
- **伺服器**：本質上是加密訊息佇列，無用戶表、無帳號概念
- **來源**：從助記詞種子短語（mnemonic seed phrase）衍生金鑰，無持久狀態、不做金鑰輪換，透過 Tor Browser 存取
- **記者**：公開發布公鑰，使用可保存狀態的本地應用，可輪換金鑰

**記者註冊**：
- 上傳長期公鑰：Diffie-Hellman 金鑰（用於訊息檢索）與 Authenticated Public Key Encryption（APKE）金鑰
- 定期上傳一次性臨時金鑰包（ephemeral key bundles）

**訊息提交**：
- 來源使用 APKE 方案加密訊息（連同自己的公鑰），提供隱式發送者認證
- 外層使用一般 PKE 加密
- 兩層加密均以 HPKE（Hybrid Public Key Encryption）實例化，提供後量子+經典混合機密性
- 訊息附帶基於 DH 的提示值（hint），讓接收者可識別訊息

**訊息檢索挑戰機制**：
- 伺服器根據 hint 值與訊息識別碼（message identifier）產生挑戰
- 將 hint 中的 Z 值以新的 DH 值重新隨機化
- 接收者利用 DH 交換律（commutativity）重新計算金鑰，解密訊息識別碼
- 回傳識別碼給伺服器以取得對應密文

**關鍵特性**：
- 來源若無待處理訊息，伺服器回應與從未使用過系統的用戶完全相同
- 無需廣播密文或嘗試解密（trial decryption）

### 安全限制與分析

已知限制：
- 來源無法輪換金鑰，發往來源的訊息缺乏前向保密
- 訊息認證為經典（非後量子）
- 訊息檢索機制為經典
- 臨時金鑰耗盡攻擊（key exhaustion）為已知開放問題

安全分析採雙管齊下：
- **計算證明（computational proofs）**：聚焦底層 PKE 方案的機密性和認證性，以及對半誠實伺服器和外部攻擊者的匿名性
- **Tamarin 符號模型證明**：由 ETH Zurich 的 Luca 和 Felix 開發，聚焦協議層的機密性和認證性

### 部署策略

根據不同組織的風險概況提供差異化部署建議：
- **大型國際新聞機構**：建議內部部署（on-premises），因具備技術和法律資源
- **小型地方新聞組織**：過去不建議使用 SecureDrop，新協議下可考慮雲端部署（外國雲端供應商可能拒絕配合當地執法）
- **獨立調查記者集體**：需逐案評估，根據相對於現有做法的風險降低程度做決策

### 後續工作

- 附件（attachment）規格設計
- 使用 hax 框架正式驗證執行期屬性的生產級 crate
- 結合 WebCAT 驗證 Tor Browser 中來源端的客戶端加密程式碼
- 專案完全開源，包含概念驗證程式碼、效能測試和 Tamarin 模型

## 與 TWCA 臺灣網路認證的關聯

1. **HPKE 混合加密的實戰應用**：SecureDrop 新協議以 HPKE 實現後量子+經典混合機密性，是 TWCA 在規劃後量子遷移路線時的重要參考案例——展示了如何在實際系統中整合混合加密而非僅做理論探討。

2. **無 CA 信任模型的極端案例**：SecureDrop 的來源端完全不依賴 CA 基礎設施，金鑰由助記詞衍生。這對 TWCA 理解某些高威脅場景下 PKI 信任鏈的局限性有啟發，也提醒 TWCA 在設計吹哨者保護或匿名舉報相關服務時需考慮不同的信任模型。

3. **元資料最小化的嚴格標準**：SecureDrop 將「用戶表」和「時間戳記」都視為不可接受的元資料洩露。TWCA 在提供電子簽章和身分驗證服務時，可反思自身系統中哪些元資料可能被不當利用，尤其在面對國家級對手時。

4. **雲端部署的安全考量**：新協議支援雲端部署，但透過端對端加密和訊息檢索挑戰機制消除對雲端供應商的信任依賴。這對 TWCA 評估雲端 HSM 或雲端憑證服務的安全架構設計具有參考價值。

5. **可否認性在真實場景中的價值**：協議刻意選擇可否認性（犧牲不可否認性），因為在某些情境下，產生歸罪證據可能危及人身安全。TWCA 在設計電子簽章服務時應注意：不可否認性雖通常被視為必要特性，但並非所有場景都適用。
