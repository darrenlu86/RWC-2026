## Abstract

利用 Certificate Transparency 大規模比對 GitHub 等平台洩漏的私鑰與公開憑證，發現近百萬把私鑰中有 42,000 把可對應至 CT 中的憑證，其中 2,622 張在 2025 年 9 月仍為有效憑證。研究揭露了私鑰洩漏後長期未被處理、撤銷機制幾乎未被使用、以及憑證擁有者對 PKI 基本概念理解不足等系統性問題，並提出 CA 層級的主動防禦建議。

## Summary

### 研究動機與方法

GitGuardian 專注於公開程式碼中的機密偵測（secret detection）。僅在 GitHub 上，2025 年就發現了 **2,900 萬個機密**（包括 API 金鑰、token、私鑰等），較 2024 年大幅成長。其中約 130,000 個是私鑰。

研究的核心問題是：這些洩漏的私鑰是否對應到真實的 TLS 憑證？攻擊者能否找到私鑰的擁有者並進行攻擊？

方法論上，從私鑰萃取公鑰，再從 CT log 中的憑證萃取公鑰，進行大規模比對（join）。理論上簡單，但實務上 CT log 規模極大——研究者估計需下載 **38 TB** 的資料。且 CT log 營運商有速率限制，部分舊 log 預計下線。因此與 Google 合作，利用其 BigQuery 資料庫進行查詢。

### 主要數據發現

- 檢視了近 **100 萬把私鑰**
- 透過 CT 比對出 **42,000 把**對應到有憑證的私鑰（大部分來自 GitHub 資料集）
- 對應到約 **36,000 個 subject**（憑證 common name，含 wildcard）
- 截至 2025 年 9 月，發現 **2,622 張仍有效的憑證**，其中 34% 的對應服務仍在公開使用中

資料集有明確偏差：私鑰可用於 TLS 以外的用途；並非所有 TLS 憑證都進入公開 CT log。

### 撤銷機制幾乎失效

研究者檢查了這些有效憑證的 CRL 和 OCSP 狀態，發現**撤銷機制幾乎未被使用**。僅極少數憑證被撤銷，其中只有 3 張的撤銷原因標記為「key compromise」。

更令人擔憂的行為模式：私鑰洩漏後，擁有者不撤銷舊憑證，而是用新憑證重新部署——導致同一服務同時存在兩張有效憑證（其中一張的私鑰已公開洩漏）。

### 長期暴露問題

利用 GitHub 的洩漏日期與憑證日期計算暴露時間，發現：
- **33% 的有效憑證**其私鑰在 2025 年 1 月之前就已洩漏
- 有 **15 把私鑰**自 2020 年洩漏至今，對應的憑證仍然有效
- 每年約有 **4,000 至 5,000 張**有效憑證的私鑰在 GitHub 上洩漏
- 許多人在私鑰洩漏後**重複使用相同私鑰**申請新憑證，而非產生新金鑰

### 漏洞揭露的困難

研究者嘗試直接聯繫憑證擁有者，出於兩個原因：避免直接強制撤銷破壞生產系統、以及提供洩漏來源等有用資訊協助修復。

透過 OSINT 技術辨識出超過 **600 個組織**，涵蓋約 1,300 張憑證。發現的組織包括：
- **19 個政府機構**（含美國政府）
- 大量 Fortune 500 企業
- **一個 CA 機構**——其自身的憑證私鑰因洩漏而被入侵

揭露結果令人失望：600 個組織中僅 **10% 回覆**。在有回覆的組織中，許多展現了對 PKI 和 TLS 基本概念的**嚴重理解不足**。典型回覆如：「該端點已使用不同憑證，之前暴露的金鑰已不在使用中，不存在可利用的場景」——因此拒絕撤銷憑證。

### 透過 CA 進行撤銷

最終研究者直接聯繫 CA，提供私鑰擁有權證明，CA 在 24 小時內完成撤銷（符合規範要求）。但此途徑的缺點是無法以隱私保護的方式向 CA 提供洩漏來源資訊。

### 改善建議

1. **CA 拒絕重複公鑰**：CA 應拒絕為已見過的公鑰簽發新憑證，防止私鑰洩漏後持續重複使用。Certbot 等自動化工具已在每次續約時自動產生新金鑰，47 天短期憑證將加速此趨勢。但有與會者指出，DANE 用戶依賴金鑰重複使用來進行 DNS TLSA 記錄與憑證的並行輪替。

2. **已洩漏金鑰的可觀測性機制**：建立一個系統讓 CA 能查詢某公鑰是否關聯到已知洩漏的私鑰，在簽發前主動拒絕。目前的問題是：私鑰今天洩漏但明天才用來申請憑證，CA 無從得知。

3. **CT 生態系統演進**：Static CT 和 CT Archive 專案使複製本研究變得更容易，但若未來憑證數量因短期憑證而爆炸，也可能使此類研究更加困難。

## 與 TWCA 臺灣網路認證的關聯

本演講對 TWCA 有**直接且實務上高度相關**的影響：

1. **私鑰洩漏偵測**：TWCA 作為 CA，應考慮建立或整合私鑰洩漏偵測機制。在收到憑證簽發請求（CSR）時，檢查其公鑰是否已出現在已知洩漏資料庫中，並拒絕為已知遭入侵的金鑰簽發憑證。這是研究者明確建議的 CA 層級改善措施。

2. **金鑰重複使用政策**：TWCA 可考慮實施政策，拒絕為已在先前憑證中使用過的公鑰簽發新憑證（至少對 Web TLS 憑證），以降低私鑰洩漏後長期暴露的風險。需注意 DANE 等用例的相容性。

3. **撤銷流程效率**：研究顯示第三方報告金鑰洩漏時的撤銷流程需要更順暢。TWCA 應確保其撤銷流程符合 24 小時內回應的業界規範，並評估是否能在撤銷流程中保護洩漏來源的隱私。

4. **客戶教育**：研究揭露的客戶對 PKI 基本概念理解不足的問題，對 TWCA 有直接啟示。TWCA 應加強客戶教育，特別是關於私鑰洩漏後必須撤銷憑證（而非僅更換憑證）的觀念，以及自動化憑證管理和金鑰輪替的最佳實踐。

5. **CT 資料的安全應用**：TWCA 可利用 CT log 資料進行主動的安全監控——不僅監控自身簽發的憑證是否被濫用，也可監控客戶域名是否出現可疑的憑證簽發。
