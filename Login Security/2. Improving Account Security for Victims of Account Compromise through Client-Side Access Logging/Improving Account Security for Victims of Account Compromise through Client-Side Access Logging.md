## Abstract

針對帳戶安全介面（ASI）易被偽冒的問題，提出客戶端加密存取日誌（CESOL）框架，設計 PKCL 和 TRACE 兩種協定，在保護使用者隱私的前提下實現不可偽冒的裝置識別，特別關注親密伴侶暴力場景中受害者偵測帳戶入侵的需求，並提出 WebAuthn 擴展方案以利實際部署。

## Summary

### 問題背景：帳戶安全介面的脆弱性

現有服務（如 Google、Signal、Meta）提供的**帳戶安全介面（Account Security Interfaces, ASIs）**讓使用者查看與帳戶關聯的裝置與工作階段。然而，2023 年的研究顯示 ASIs 極易被偽冒——僅需使用瀏覽器開發者工具修改 **User-Agent 字串**，攻擊者即可讓自己的登入記錄偽裝成受害者已知的裝置。

### 親密伴侶暴力的特殊情境

講者 Carolina Ortega Pérez 同時是紐約「終結科技濫用診所（Clinic to End Tech Abuse）」的志工，協助親密伴侶暴力（Intimate Partner Violence, IPV）倖存者處理科技安全問題。帳戶入侵是倖存者最常面臨的問題：

- 施暴者通常知道密碼或能回答安全問題
- 在某些情境下，移除施暴者的存取權限可能導致**人身安全風險**，因此單純更改密碼未必可行
- 倖存者需要能**可靠偵測**帳戶是否被入侵的機制

### 為何不能用靜態裝置識別碼

直覺的解決方案是在登入時傳送靜態裝置識別碼（如序號），但多年隱私研究已證明**裝置指紋辨識（device fingerprinting）**的危害——伺服器可藉此追蹤使用者。Android 最佳實踐也建議「盡可能使用可重設的識別碼」。因此需要**作業系統層級的支援**，在不向伺服器洩露靜態識別碼的前提下實現裝置辨識。

### 協定一：PKCL（Public Key CESOL）

**架構**：
- 作業系統（OS）持有裝置識別碼並產生公私鑰對，假設 OS 為可信任的
- 登入時，OS 用自己的公鑰加密裝置識別碼，連同公鑰一起傳送給伺服器
- 伺服器儲存密文與公鑰，但無法解密
- 當新裝置登入時，伺服器發送此帳戶所有已知公鑰，新裝置用所有公鑰加密自己的裝置識別碼
- 舊裝置查看 ASI 時，伺服器回傳以該裝置公鑰加密的所有密文，裝置可解密查看所有登入記錄

**Gossip 協定**：後加入的工作階段可透過先前工作階段的協助取得舊密文的存取權。

**限制**：
- 若舊工作階段不在線上，新工作階段無法解密舊的日誌條目（**不完整性**）
- 僅能抵禦**誠實但好奇的伺服器（honest-but-curious server）**，無法防禦惡意伺服器注入幽靈使用者（ghost users）——伺服器可注入自己的公鑰，讓使用者用惡意公鑰加密裝置識別碼

### 協定二：TRACE（基於同步金鑰的 CESOL）

**核心概念**：引入一把在所有使用者裝置間同步的對稱金鑰 K，用以加密裝置識別碼。

**同步方式一——基於密碼與 OPRF**：
- 引入**Oblivious PRF（OPRF）服務**，持有秘密金鑰 SK
- 使用者的 OS 與 OPRF 服務互動：發送隨機化請求 → OPRF 盲目評估 → 回傳結果
- OS 由請求與回應推導出金鑰 K = PRF(SK, password)
- OPRF 服務不得知密碼或金鑰，且推導為確定性——不同裝置用相同密碼會得到相同金鑰 K
- 建議部署**多個 OPRF 服務**以增強抗攻陷能力

**架構調整**：加密後的裝置識別碼儲存於 OPRF 服務而非原始服務提供者（RP），使 RP **無需知道 TRACE 是否在運行**，提升向後相容性。

**強制留痕機制**：為防止惡意使用者繞過日誌記錄，將**憑證推導移至 OS 層**——OS 先與 OPRF 服務互動並儲存加密的裝置識別碼，**之後才將登入憑證交給應用程式**。如此一來，每次認證都會自動留下不可繞過的痕跡。

**同步方式二——WebAuthn CESOL 擴展**：
- 利用 WebAuthn 憑證（如 Passkeys）的同步機制
- 提出新的 **CESOL 擴展**，類似現有的 PRF 擴展，在秘密金鑰上評估 PRF 並以不同的 domain separator 做隔離
- 關鍵差異：PRF 擴展的輸出交給客戶端應用程式，而 **CESOL 擴展的輸出僅交給 OS / CESOL 服務**，確保應用程式無法取得解密金鑰
- 藉由 Passkeys 的跨裝置同步能力自動實現金鑰同步，使用者無需記憶額外密碼

### 兩種協定的比較

| 特性 | PKCL | TRACE |
|------|------|-------|
| 不可偽冒的 ASI | 是 | 是 |
| 持久狀態需求 | 無 | 需要密碼或同步金鑰 |
| 日誌完整性 | 部分情況不完整 | 完整 |
| 隱私保證 | 誠實但好奇的伺服器 | 惡意伺服器 |
| RP 需知情 | 是 | 否（可向後相容） |

### 部署考量

部署的核心瓶頸是**作業系統廠商的支援**。若能說服 Apple、Google、Microsoft 支援 CESOL，即可覆蓋全球約 **95% 的使用者**。講者指出 Passkeys 的推動經驗已證明這種跨平台合作是可行的。研究團隊已實作兩個協定的概念驗證，效能不構成問題。

## 與 TWCA 臺灣網路認證的關聯

本場演講對 TWCA 有多方面的啟示：

1. **帳戶安全介面的可信度**：TWCA 提供的各類線上服務（如憑證管理入口）同樣仰賴工作階段管理與登入紀錄。CESOL 框架揭示了 User-Agent 易被偽冒的根本問題，TWCA 應評估自身帳戶安全介面是否存在類似風險，並考慮採用加密日誌方案增強可信度。

2. **WebAuthn / Passkeys 整合機會**：TRACE 協定的 WebAuthn CESOL 擴展提出了一種在 Passkeys 基礎設施上增加隱私保護裝置識別的方法。TWCA 若在其身分驗證服務中整合 Passkeys，可同時評估納入此類擴展，為企業客戶提供更強的帳戶入侵偵測能力。

3. **OPRF 服務的營運角色**：TRACE 協定需要獨立的 OPRF 服務，且建議部署多個以分散信任。TWCA 作為受信任的第三方機構，具備營運此類 OPRF 服務的天然定位，可作為密碼管理器或身分驗證服務的金鑰推導基礎設施。

4. **親密伴侶暴力與數位安全**：此研究凸顯密碼學技術在保護弱勢群體方面的應用價值。TWCA 在推動電子簽章與身分驗證服務時，應將此類使用情境納入安全設計考量，確保系統不僅服務一般使用者，也能為高風險使用者提供有效的安全保障。
