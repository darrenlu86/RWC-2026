## Abstract

Nadia Heninger 教授透過在 UC San Diego 計算機科學大樓屋頂架設一座一米口徑的現成衛星天線，對數百顆地球同步衛星（geostationary satellites）的轉頻器進行系統性被動監聽，意外揭露了學術研究者通常無法觸及的內部網路中，密碼學實踐的驚人落後。研究團隊在 CCS 2025 發表的論文主要聚焦於未加密資料，而本次 RWC 2026 受邀演講則重新分析資料集並補充新的資料收集，從密碼學實踐的角度呈現衛星轉頻器上所觀察到的加密現況：從設計有瑕疵的衛星加密標準（DVB-CSA），到使用 null cipher 的 IPSec 隧道、明文廣播的軍事座標、未受保護的工業控制系統，以及 HTTPS Everywhere 在公共瀏覽器流量上的成功與在關鍵基礎設施上的全面失敗。

## Summary

### 研究背景與動機

Nadia Heninger 回顧她自 2013 年第二屆 RWC 以來的研究歷程——當年她在發燒 101 度的狀態下發表了 TLS 弱 RSA 金鑰的研究。她指出，從那時起 RWC 社群深入研究 TLS 並發現一系列攻擊、推動修復，使 HTTPS 基礎設施達到了相當好的狀態。然而，**學術研究者有能見度的基礎設施（如公開的 HTTPS）已趨完善，但在研究者無法觸及的網際網路角落，隱藏著大量未知的安全災難**（"untold horrors lying around in the parts of the internet that we have no visibility into"）。

本研究源自她與無線電領域同事 Aaron Shulman 的腦力激盪。選擇衛星基礎設施的原因有三：(1) 充斥著遺留硬體（legacy hardware），幾乎不可能更新；(2) 設計思維可能停留在 1990 年代甚至更早；(3) 幾乎沒有學術研究者在關注這個領域。這是一個未獲資助的研究計畫，所有設備費用來自 Heninger 的自由裁量經費。

### 地球同步衛星基礎知識

研究聚焦於**地球同步衛星（GEO）**，排除低軌道（LEO）與中軌道（MEO）衛星（如 Starlink）。GEO 衛星的特性：

- 固定在赤道上方的軌道位置，數百顆
- 每顆衛星可有數十個轉頻器，使用不同頻率與極化
- 每個轉頻器的波束覆蓋範圍從單一地面站到整個大陸不等（可在 Satbeams 網站查看）
- GEO 提供 100% 可靠性（不同於 LEO 的 9-10 分鐘可見窗口與覆蓋空隙）
- 大量關鍵應用仍依賴 GEO，且有大量尚未遷移的遺留基礎設施

衛星本身僅作為**「彎管」（bent pipe）**——在使用者終端與地面站之間轉發訊號。實際的協定棧與加密完全由終端設備決定，且往往是專有的、無公開文件的廠商特定實作。

### 資料收集方法與法律倫理

硬體設置全部使用**現成設備**：
- 一米口徑衛星天線 + 馬達支架（可沿軌道弧擺動）
- 現成衛星解碼器與 TV 調諧卡（研究公開後該型號調諧卡售罄且停產）
- 筆電執行開源衛星訊號解碼軟體（業餘愛好者社群使用的工具）

資料收集方式是**緩慢掃描整個天空**——天線一次只能對準一個方向，調整接收頻率與極化後，發現訊號即停留接收，再繼續掃描。研究團隊嚴格遵守三項原則：**只接收不發送、不嘗試解密任何資料**，且事先諮詢了 UC 總法律顧問辦公室。Heninger 指出各國對接收無線電與衛星通訊的法律差異極大，使得此類研究在其他地方難以複製。

由於 San Diego 位於美墨邊境，團隊意外地接收到大量針對墨西哥廣播的衛星訊號——這並非刻意挑選墨西哥，純粹是地理位置所致。

### 協定棧逆向工程

研究生花了**數年時間**逆向工程各種專有協定棧。早期衛星協定以影片傳輸為導向（資料嵌入 MPEG 串流），較新協定才面向 IP 資料。協定棧包含物理層、鏈路層與 IP 資料層，但實際邊界模糊。團隊盡最大努力（best effort）解析所有能解析的內容——僅執行 `strings` 指令就能看到大量明文，完整重建網路層後更為驚人。

**在此層看到明文，代表系統在所有可能的加密層級上都失敗了**——衛星傳輸協定層、網路層（IPSec）、應用層（TLS）全部未加密。

### 衛星加密演算法的設計缺陷

**DVB-CSA1（1990 年代）**：
- 自訂區塊密碼 + 自訂串流密碼的組合
- 實際金鑰強度僅 **48 位元**
- 已有碩士論文用 FPGA 實現破解
- 至今仍是**預設的加擾演算法**，但主要用於付費電視

**DVB-CSA3（2013 年採納）**：
- 基於 AES，但設計令人震驚：
  - 使用 **CBC 模式**，IV 為規格書中硬編碼的**常數**（該 IV 的 ASCII 解碼包含制定者的姓名彩蛋）
  - 不足 16 bytes 的尾端資料**直接明文傳輸**（規格書明確寫明 "it stays in the clear"）
  - 2013 年的標準卻不使用已存在多年的認證加密（AEAD）
- 實際部署緩慢，因需要新硬體，僅在少數轉頻器上觀察到
- Heninger 諷刺這像是 CTF 競賽的題目而非正式標準

**DVB-S2 GSE（Generic Stream Encapsulation）層**：
- 標頭有加密旗標，但加密演算法**未標準化**，為廠商特定實作
- 可嘗試逆向工程或閱讀 FIPS 文件來推測演算法
- 大部分串流根本未加密，「直接看明文就好」

### 流量分類與 IPSec 觀察

衛星轉頻器上觀察到的流量類型：消費者 ISP、企業網路、工業控制系統（ICS）、多家機上 Wi-Fi 供應商、船舶、政府流量、手機基地台回傳、VoIP、衛星網路自身管理流量等。

少數轉頻器使用了 IPSec 隧道進行全面加密。但分析交握過程中的密碼套件選擇，發現這些設備**停留在 2010 年的時空**：
- SHA-1 碰撞？不存在的
- LogJam 攻擊？從未發生
- 使用預共享金鑰（PSK），密碼品質令人擔憂

Heninger 給予評價：「至少有加密，沒有明文廣播憑證，**B+**」。由於是長期存活的 VPN 隧道，交握極少發生，可用的元資料有限。

### 手機基地台回傳（Cell Tower Backhaul）

Heninger 在研究前完全不知道有手機基地台使用衛星進行網路回傳。這發生在極端偏遠的山區或沙漠地區，無法透過光纖或微波連線。衛星回傳的內容包括：
- **使用者資料**：通話音訊、簡訊內容、上網流量
- **控制平面資料**：session keys、handover metadata、IMSI 等

最嚴重的案例：**T-Mobile** 數座基地台的 IPSec 隧道使用了 **null cipher**——它們確實建立了 IPSec 連線，但密碼演算法設為空，所有內容以明文廣播至整個衛星覆蓋範圍。T-Mobile 安全團隊在收到通報後**立即修復**，表現專業。根本原因是使用了外部供應商，T-Mobile 對自身網路配置缺乏能見度。其他電信業者的回應速度則參差不齊，有些至今仍難以聯繫。

### 機上 Wi-Fi

機上 Wi-Fi **完全未加密**，可觀察到乘客瀏覽流量與機組人員端點（如 ForeFlight 飛行規劃工具、座椅背娛樂系統）。此問題數年前已有人向供應商通報，但他們選擇不修復，本次揭露後依然如此。

航空公司的安全政策將其定位為「咖啡廳 Wi-Fi」——Heninger 嘲諷道：「如果你的咖啡廳覆蓋整個大陸的話」。好消息是 HTTPS Everywhere 使大部分使用者流量有加密保護。此外，許多航空公司正遷移至 **Starlink**（已加密），可能因此不再投資修復遺留系統。

### RSA 金鑰復原問題

在傳輸資料中發現了部分 RSA 私鑰——僅含 PKCS#1 DER 編碼格式末端的欄位（Q^{-1} mod P、D mod Q-1、部分 D mod P-1），但**不含模數 N 和公鑰指數 E**。Heninger 稱這是「進階版的 fuck-a-duck 問題」（一個已知的密碼學趣味挑戰），共同作者 Keegan Ryan（出席了本次 RWC）設計了優美的數學解法，收錄於論文擴展版附錄。Heninger 鼓勵學生「不要發明 CTF 題目——世界本身就是你的 CTF」。

### TLS 現況：令人欣慰

更新的資料顯示 TLS 流量狀況**良好**：
- 可觀察到 **TLS 1.3** 的普及
- 可觀察到**後量子密碼學（PQC）**的實際部署
- 異常密碼套件的使用率不到 5%
- **HTTPS Everywhere 對公共網路流量是成功的**

### DNS：幾乎完全裸露

DNS 流量與 TLS 的成功形成鮮明對比：
- 絕大多數 DNS 流量**完全未加密且未認證**
- DNS over TLS 使用量極微（根據發送至知名供應商的封包數判斷）
- DNSSEC 有少量部署但遠不普及

### 政府與軍事流量

**墨西哥政府與軍事**：大量明文傳輸，包括黑鷹直升機的**精確追蹤座標**等元資料。部分流量在揭露後已消失，部分仍然存在。向外國軍方進行漏洞揭露被證明極其困難。

**美國政府**：大部分流量以 IPSec 隧道謹慎保護，但仍有來自**船艦與海軍基地的未加密 VoIP 流量**——顯然是衛星供應商的配置意外。在研究公開後，美國政府非常認真地對待並修復了問題，但前期聯繫正確的負責人也非常困難。

### 網路協定災難全覽

- **BGP 路由協定**：大量未認證的 BGP 流量被廣播
- **SNMP community strings**（等同管理密碼）：明文傳輸
- **Telnet**：密碼可見，同時觀察到大量暴力破解攻擊嘗試
- **RADIUS 認證協定**：在衛星轉頻器上發現（直接反駁審稿人「RADIUS 攻擊不具意義因為都在安全網路上」的評論）
- **工業控制系統（ICS）**：完全未受保護地廣播，Heninger 表示「這實在令人恐懼」，監管機構對此似乎沒有規範
- **未加密 VoIP**：來自墨西哥電信業者 Telmex 的通話流量

### 威脅模型的重新思考

Heninger 坦言，過去她的威脅模型關注的是「NSA 竊聽光纖電纜」之類的場景，從未想過**某些供應商可能正在將我們的網路流量向全球廣播**。更關鍵的是，作為終端使用者，我們**無從得知**自己的手機基地台是用什麼方式連接到網路其餘部分的。

### 結論：需要「HTTPS Everywhere for the Rest of the Internet」

核心發現的矛盾：HTTPS Everywhere 對終端使用者瀏覽器流量是成功的，**但對電廠、軍隊和關鍵基礎設施卻全面失敗**。即使高度重視安全的美國政府，也因複雜的多層客戶—供應商關係而對自身網路配置缺乏能見度。

結構性挑戰：
- 在某些應用場景中，**連線性確實比安全性更重要**（如從荒野撥打 911，通話能接通比通話隱私更重要）
- 組織對自身網路配置缺乏能見度，即使**有意啟用加密也可能因配置錯誤而失效**
- 私有網路中「全面加密」的運動已有一些進展（本屆 RWC 也有相關演講），但尚未滲透到所有產業
- 學術研究者**無法合乎倫理地掃描內部網路**，因此目前沒有明確的實現路徑

## 與 TWCA 臺灣網路認證的關聯

本演講對 TWCA 有多面向的重要啟示：

1. **Web PKI 以外的加密盲區**：TWCA 作為憑證機構，核心業務在公開 Web PKI 中的 HTTPS 憑證簽發。本研究明確揭示 HTTPS 的成功僅限於公共瀏覽器流量——企業內部網路（衛星回傳、ICS/SCADA、VPN 隧道）中的加密缺失極為嚴重。TWCA 為企業客戶提供 SSL/TLS 憑證服務時，可主動提醒客戶審視其非 Web 通訊（如衛星鏈路、專用網路、IoT 設備）的加密狀況，將安全諮詢服務延伸至 PKI 以外的領域。

2. **Null Cipher 與配置錯誤的教訓**：T-Mobile 的 IPSec null cipher 案例顯示，「有部署加密」不等於「有效加密」。TWCA 在推廣 TLS 憑證時，可強調**配置驗證**的重要性——不只是安裝憑證，更要確認密碼套件選擇、協定版本、金鑰長度等是否符合當前安全標準。TWCA 的稽核與合規服務可納入 TLS 配置檢測（如禁用 null cipher、弱密碼套件、過時協定版本）。

3. **DNS 安全與憑證簽發流程**：DNS 幾乎完全未加密的現況，對依賴 DNS 進行域名驗證（Domain Control Validation, DCV）的 CA 業務構成直接威脅。在衛星網路等場景中，DNS 查詢可被任何擁有衛星天線的人觀察甚至竄改。TWCA 應積極推動 DNS over TLS/HTTPS 與 DNSSEC 的採用，並在憑證簽發流程中採用多路徑驗證（multi-perspective validation）以降低 DNS 劫持風險。

4. **供應鏈可見性與第三方風險管理**：T-Mobile 因外部供應商配置錯誤而暴露流量、美國軍方因多層供應商關係而不知自身網路有明文傳輸——這凸顯了**供應鏈安全**的關鍵性。TWCA 可在其資安服務中強調：客戶應定期稽核其通訊供應商（ISP、衛星服務商、VPN 供應商）的加密配置，而非僅信賴「已啟用加密」的聲明。

5. **後量子密碼學遷移的正面訊號**：TLS 流量中已可觀察到後量子密碼學（PQC）的實際部署，這對 TWCA 規劃 PQC 遷移時程是正面的參考指標。然而衛星基礎設施中仍大量使用 1990 年代的演算法（48-bit DVB-CSA1），顯示非 Web 領域的密碼學遷移將遠比 Web PKI 更加緩慢和困難。

6. **遺留基礎設施中的密碼學標準問題**：DVB-CSA3 在 2013 年仍使用常數 IV 的 AES-CBC 而非 AEAD，證明即使是「新」標準也可能內建密碼學缺陷。TWCA 參與或制定臺灣相關電子簽章、身分驗證標準時，應確保採用現代密碼學最佳實踐，避免重蹈覆轍。
